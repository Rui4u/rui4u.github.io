Mach [mʌk]

步骤

创建了两个线程

  1. 主异常处理线程

  2. 次要异常处理线程。（先暂停，主异常处理异常后，再回复次要异常处理线程）
    为啥创建俩，防止主要异常处理线程崩溃。

  3. 通过 mach_task_self() 获取当前进程对应的 task。

  4. 通过 task_get_exception_ports() 获取原本处理异常的 port，并保存在 g_previousExceptionPorts 中。

  5. 通过 mach_port_allocate() 创建新的异常处理端口。

  6. 通过 mach_port_insert_right() 给这个新创建的端口申请权限

  7. 通过 task_set_exception_ports() 把异常接收的 port 设置为自己新创建的 port

  8. 创建好 port 之后，就可以创建自己的线程去一直读取 port 上的消息了。通过 pthread_create() 创建自己的线程，以及设置好执行的方法为 handleExceptions。

信号量Signal

分类

  1. SIGABRT (Abort):
    • 用于表示程序调用了 abort() 函数，通常是在出现严重错误时强制终止程序。
    • 可能由应用程序自身或系统发起。
  2. SIGBUS:
    • 表示非法内存访问，如对非法地址进行读写操作。
    • 可能由硬件错误或编程错误引起。
  3. SIGFPE (Floating-Point Exception):
    • 表示浮点运算异常，如除以零或计算结果无穷大。
    • 通常与浮点数计算相关。
  4. SIGILL (Illegal Instruction):
    • 表示执行了非法或未定义的指令。
    • 可能由二进制文件损坏或使用错误的机器码引起。
  5. SIGPIPE:
    • 表示管道或套接字被关闭，但仍然进行写操作。
    • 通常发生在使用网络或进程间通信时。
  6. SIGSEGV (Segmentation Violation):
    • 表示非法内存访问，如空指针访问、已释放内存访问等。
    • 是一种常见的崩溃类型。
  7. SIGSYS (Bad System Call):
    • 表示执行了未定义的系统调用。
    • 通常与底层系统相关。
  8. SIGTRAP (Trap):
    • 表示触发了断点或调试陷阱。
    • 在调试过程中使用。

信号量捕捉的两种方式

  1. void(*signal(int, void (*)(int)))(int);
  2. int sigaction(int signum, const struct sigaction *act, struct sigaction *oldact);

两者的区别：
int sigaction():

  1. int sigaction()方式会传递信号处理方式的结构体，会传入一个已经开辟的空间，处理异常的操作空间先初始化，避免递归问题无法创建空间上传的问题
  2. action.sa_flags = SA_SIGINFO; 可以获取崩溃信息， 例如崩溃地址，信号量名称，信号量code等。 返回回调类型为siginfo_t;

问题

为什么捕捉Mach异常还需要捕捉UNIX信号:

在 macOS 和 iOS 等系统中，Mach 异常和 UNIX 信号是两种不同的异常处理机制，分别用于处理不同类型的异常。

  1. Mach 异常：
    • Mach 是 macOS 和 iOS 等系统的内核，它负责管理进程间通信和系统调用等底层操作。
    • Mach 异常是由内核发出的，用于处理诸如内存访问错误、非法指令等底层异常。
    • Mach 异常处理通过使用 Mach 线程异常处理器来捕捉和处理异常。可以使用 task_set_exception_ports() 函数设置 Mach 异常的处理回调函数。
  2. UNIX 信号：
    • UNIX 信号是用于进程间通信和处理高层次事件的机制，例如用户按下 Ctrl + C 时产生的 SIGINT 信号。
    • UNIX 信号可以用于捕捉和处理一些高级事件，如程序终止请求、键盘中断等。
    • UNIX 信号处理通过使用 signal() 或 sigaction() 函数来设置信号处理器回调函数。

由于 Mach 异常处理机制是底层的异常处理机制，而 UNIX 信号是高层的进程间通信和事件处理机制，因此在实际应用中，捕捉 Mach 异常和 UNIX 信号可以用于处理不同类型的异常和事件。一些应用场景可能需要同时处理底层的 Mach 异常和高级的 UNIX 信号。

例如，在一个应用程序中，可能需要捕捉 Mach 异常来处理底层的内存访问错误或非法指令，以确保程序在出现底层异常时可以正常退出或处理异常。同时，也可能需要捕捉 UNIX 信号来处理用户的交互操作，如终止应用程序或执行特定操作等。

因此，捕捉 Mach 异常和 UNIX 信号是为了实现全面的异常处理和事件处理，使得应用程序能够在出现各种异常和事件时做出相应的处理和响应。同时，需要注意两种处理机制的使用方式和区别，以确保异常处理和事件处理的准确性和稳定性。

首先要明确的一点是，Mach异常和UNIX信号都可以被捕获，他们也几乎一一对应，但我们需要优先处理Mach异常，因为Mach异常更接近底层，而UNIX信号存在被Mach默认异常处理函数直接退出进程而无法生成的情况。

有一些异常如EXC_CRASH，在Mach异常的阶段没有捕捉，而是放到UNIX信号中捕捉，其中的解释在PLCrashReporter的注释中有详细的解释：

UNIX信号

Mach已经通过异常机制提供了底层的异常处理，但为了兼容更为流行的POSIX标准，BSD在Mach异常机制之上构建的UNIX信号处理机制。异常信号首先被转换为Mach异常，如果没有被外界捕捉，则会被默认的异常处理ux_exception()转换为UNIX信号。
例如SIGSEGV（非法内存访问）转换为 EXC_BAD_ACCESS

  1. 为什么要挂起线程：
    在打印线程信息的时候，线程是动态变化的。在捕捉后挂起线程，是为了保证当前信息是当前时刻的状态。
    挂起线程后需要注意死锁线程同步问题，可能造成的死锁问题。 同时结束后也需要恢复线程。

    1. 线程安全问题
    2. 阻塞其他操作
